# `yosh-plugin update`: structural TOML rewrite via `toml_edit`

**Date:** 2026-04-28
**Scope:** `crates/yosh-plugin-manager`
**Tracks TODO item:** "`yosh-plugin update`: version replacement uses naive `String::replacen` which may target wrong plugin if two share the same version" (TODO.md, Plugin System Enhancements)

## Problem

`cmd_update` in `crates/yosh-plugin-manager/src/lib.rs:134-194` rewrites
`plugins.toml` by string substitution:

```rust
new_content = new_content.replacen(
    &format!("version = \"{}\"", current),
    &format!("version = \"{}\"", latest),
    1,
);
```

This breaks correctness in two cases:

1. **Same-version siblings.** If two `[[plugin]]` entries share a `version`
   string, `replacen(_, _, 1)` rewrites the *first* occurrence in document
   order, which may belong to a plugin that was not the update target. The
   user's manifest is silently corrupted.
2. **Token collisions outside `[[plugin]]`.** Any `version = "X"` literal
   elsewhere in the file (a stray comment that includes the literal, a
   future top-level field) is a candidate for accidental rewrite.

The fix is to address plugins by their `name` key and to mutate the TOML
through `toml_edit::DocumentMut` (already a dependency, used identically
in `src/install.rs`). Name uniqueness is *conventional*
(`config::validate_plugin_name` validates characters but not uniqueness),
so the helper detects duplicate names defensively and errors rather than
silently rewriting the first match ‚Äî closing the same class of
"silently-target-the-wrong-entry" bug at the new layer.

## Goals

- Rewrite `version` only on the `[[plugin]]` table whose `name` matches
  the update target.
- Preserve comments, blank lines, key ordering, and quote style.
- Parse `plugins.toml` once per `update` invocation; write at most once.
- Match the crate's existing module convention (`install.rs`, `sync.rs`,
  `verify.rs`) by extracting the command's business logic into
  `src/update.rs`.

## Non-goals

- Changing `cmd_sync`'s post-update lockfile regeneration (still invoked
  on `any_updated`).
- Adding `--dry-run`, `--allow-downgrade`, or version pinning options
  (out of scope; potential follow-ups).
- Touching `plugins.lock` directly from `update.rs` (lockfile is
  regenerated by `cmd_sync`).

## Invariants

1. **One parse, one write.** `DocumentMut` is parsed exactly once at the
   start; `fs::write` is invoked at most once at the end (and only if
   `any_updated`). All in-loop mutations operate on the in-memory
   document.
2. **Lookup by `name`.** `set_plugin_version` walks the
   `ArrayOfTables` under `doc["plugin"]` and matches each table's
   `name` field literally. Name uniqueness is conventional, not
   enforced by `config::load_config`; the helper therefore counts
   matches and errors if more than one is found (see Error
   Handling). The old version literal is never read.
3. **No-op write avoidance.** If every plugin is `AlreadyLatest`,
   `update` returns without touching the file (mtime stable).
4. **`cmd_sync` invocation parity.** Behavior matches HEAD: `cmd_sync(false)`
   runs only when at least one plugin was updated.

## Error Handling

Two layers:

| Layer | Type | Examples | Behavior |
|---|---|---|---|
| Fatal (config I/O) | `Err(String)` from `update()` | config read/write failure, TOML parse failure, target plugin missing in TOML | `cmd_update` exits 2. |
| Per-plugin | `UpdateStatus::Failed(String)` | GitHub API failure (rate limit, network, 404) | Loop continues. Result printed per-plugin. |

`set_plugin_version` returns `Err` (Fatal) for:

- `doc["plugin"]` absent or not `ArrayOfTables` ‚Üí
  `"config 'plugin' key is not an array of tables"`
- No table found whose `name` matches ‚Üí `"plugin '<name>' not found in config"`
  (defensive: should not occur because `config::load_config` succeeded immediately before)
- More than one table whose `name` matches ‚Üí `"plugin '<name>' appears multiple times in config"`
  (defensive: name uniqueness is conventional, not enforced; the helper refuses to guess which entry to mutate rather than silently rewriting the first one)

Error string style follows `install::write_plugin_entry`. The
`yosh-plugin: ` prefix is added by `cmd_update`, not by `update.rs`.

**Partial-failure persistence (preserved behavior):** if some plugins
update successfully and others fail their GitHub call, the successful
mutations are persisted on the final `fs::write`. This matches the
current `String::replacen` flow (which writes `new_content` once after
the loop).

## Compatibility Break vs HEAD

None observable to users in successful paths. The textual diff produced
by `toml_edit` may differ from `String::replacen` in formatting edge
cases:

- `version = "X"` written by `toml_edit::value(...)` uses the standard
  basic-string form with single space around `=`, matching what
  `install::write_plugin_entry` already produces.
- `toml_edit` preserves surrounding trivia (comments, blank lines,
  ordering) verbatim, so manifest layout outside the touched value is
  unchanged.
