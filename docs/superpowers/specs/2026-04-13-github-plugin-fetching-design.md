# GitHub Plugin Fetching — Design Spec

## Overview

Add the ability to fetch prebuilt plugin binaries from GitHub Releases, managed through a declarative `plugins.toml` and a lock file workflow via a separate `kish-plugin` CLI tool.

## Goals

- **Security**: HTTPS + SHA-256 checksum verification + existing capability sandbox
- **Extensibility**: Asset template system for flexible binary naming conventions
- **Performance**: kish binary stays lightweight; network logic lives in a separate tool

## Non-Goals

- Source-from-build compilation
- Cryptographic signature verification (may be added later)
- Plugin registry / central index
- GitHub Enterprise support (initial version)

---

## Configuration Format

### plugins.toml (User-Edited)

```toml
# GitHub plugin
[[plugin]]
name = "git-status"
source = "github:user/kish-plugin-git-status"
version = "1.2.3"
enabled = true                                      # Default: true, optional
capabilities = ["variables:read", "io", "hooks:pre_prompt"]

# GitHub plugin with custom asset template
[[plugin]]
name = "auto-env"
source = "github:other/kish-plugin-auto-env"
version = "0.5.0"
asset = "kish_auto_env-{os}-{arch}.{ext}"           # Optional, overrides default
capabilities = ["variables:read", "variables:write", "filesystem"]

# Local plugin
[[plugin]]
name = "local-tool"
source = "local:~/.kish/plugins/liblocal.dylib"
enabled = true
capabilities = ["io"]
```

### plugins.lock (Auto-Generated)

```toml
[[plugin]]
name = "git-status"
path = "~/.kish/plugins/git-status/libkish_git_status.dylib"
enabled = true
capabilities = ["variables:read", "io", "hooks:pre_prompt"]
sha256 = "a1b2c3d4e5f6..."
source = "github:user/kish-plugin-git-status"
version = "1.2.3"

[[plugin]]
name = "local-tool"
path = "~/.kish/plugins/liblocal.dylib"
enabled = true
capabilities = ["io"]
sha256 = "f6e5d4c3b2a1..."
source = "local:~/.kish/plugins/liblocal.dylib"
```

### kish Binary Loading Logic

- kish reads **only** `plugins.lock`. It never reads `plugins.toml`.
- If `plugins.lock` does not exist, kish starts with zero plugins.
- The lock file entry fields `path`, `enabled`, and `capabilities` are used for loading. The `sha256`, `source`, and `version` fields are informational and ignored by kish at load time.

## Asset Template System

### Default Template

```
lib{name}-{os}-{arch}.{ext}
```

### Template Variables

| Variable | Description | Example Values |
|----------|-------------|----------------|
| `{name}` | Plugin name with hyphens converted to underscores | `git_status` |
| `{os}` | Operating system | `linux`, `macos` |
| `{arch}` | CPU architecture | `x86_64`, `aarch64` |
| `{ext}` | Shared library extension | `so` (Linux), `dylib` (macOS) |

### Resolution Example

Plugin `git-status` on macOS aarch64:

- Default: `lib{name}-{os}-{arch}.{ext}` → `libgit_status-macos-aarch64.dylib`
- Custom: `kish_git-{os}-{arch}.{ext}` → `kish_git-macos-aarch64.dylib`

## `kish-plugin` CLI

### Subcommands

#### `kish-plugin sync`

1. Parse `plugins.toml`
2. Read existing `plugins.lock` if present; compute diff
3. For new GitHub plugins:
   - Call GitHub API: `GET /repos/{owner}/{repo}/releases/tags/v{version}`
   - Fall back to tag `{version}` (without `v` prefix) if not found
   - Match asset name against resolved template
   - Download binary via `browser_download_url`
   - Compute SHA-256
   - Place in `~/.kish/plugins/{name}/`
4. For new local plugins:
   - Verify file exists at resolved path
   - Compute SHA-256
5. For removed plugins (in lock but not in toml):
   - Remove from lock file
   - Binary files are kept on disk (use `--prune` to delete)
6. For unchanged plugins (same version, already in lock):
   - Verify local binary SHA-256 matches lock file
   - If mismatch: warn and re-download
   - If re-downloaded SHA-256 also differs from lock: report as error (asset may have been replaced upstream)
7. Write lock file with all successful entries
8. Report failures to stderr
9. Disabled plugins (`enabled = false`): download and record in lock file with `enabled = false`, but kish will not load them
10. Exit code: 0 = all success, 1 = partial failure, 2 = fatal error

#### `kish-plugin update [name]`

1. Query GitHub API for latest release tag
2. Update `version` in `plugins.toml` (note: TOML rewriting may strip comments; users should be aware of this limitation)
3. Run sync flow for affected plugins
4. If `name` is omitted, update all GitHub plugins

#### `kish-plugin list`

Display lock file contents in tabular format:

```
git-status    1.2.3  github:user/kish-plugin-git-status     ✓ verified
auto-env      0.5.0  github:other/kish-plugin-auto-env      ✓ verified
local-tool    -      local:~/.kish/plugins/liblocal.dylib    ✓ verified
```

#### `kish-plugin verify`

- Compute SHA-256 of every binary referenced in lock file
- Compare against recorded checksum
- Report mismatches and missing files
- Exit code: 0 = all match, 1 = mismatch or missing

## GitHub API Integration

### Authentication

- **No token**: Public repositories only. Rate limit: 60 requests/hour.
- **With token**: Set `YOSH_GITHUB_TOKEN` or `GITHUB_TOKEN` environment variable. Sent as `Authorization: Bearer {token}`. Supports private repositories. Rate limit: 5,000 requests/hour.

### API Usage

- Base URL: `https://api.github.com`
- Release lookup: `GET /repos/{owner}/{repo}/releases/tags/{tag}`
- Tag resolution: try `v{version}` first, fall back to `{version}`
- Response parsing: extract `assets[].name` and `assets[].browser_download_url`
- HTTPS only; plain HTTP is rejected

## Security Model

### Multi-Layer Defense

```
Layer 1: HTTPS transport (TLS verification via HTTP client defaults)
    ↓
Layer 2: SHA-256 checksum (lock file, TOFU model)
    ↓
Layer 3: Capability sandbox (existing, unchanged)
```

### Layer 1: HTTPS

- All GitHub API calls and downloads use HTTPS exclusively
- TLS verification delegated to HTTP client library (`ureq`)

### Layer 2: SHA-256 Checksum

**First sync (TOFU):**
- Download binary, compute SHA-256, record in lock file
- First download is trusted

**Subsequent syncs (same version):**
- Verify local binary against lock file checksum
- Match → skip (no re-download)
- Mismatch → warn, re-download. If re-downloaded hash also differs from lock → error (possible upstream asset replacement)

**`kish-plugin verify`:**
- Verify all entries. Non-zero exit on any mismatch.

### Layer 3: Capability Sandbox (Existing)

- `capabilities` in `plugins.toml` defines allowed capabilities per plugin
- kish intersects declared ∩ configured capabilities at load time
- Denied API calls are rejected via stub functions with stderr logging
- No changes to existing sandbox implementation

### Lock File Git Management

Committing `plugins.lock` to version control is recommended. This ensures:
- Team members and other machines reproduce the same verified binaries
- `kish-plugin sync` validates downloaded binaries against committed checksums

## Crate Structure

### New Crate: `crates/kish-plugin-manager/`

```
crates/kish-plugin-manager/
├── Cargo.toml
└── src/
    ├── main.rs          # CLI entry point (subcommand dispatch)
    ├── config.rs         # plugins.toml parsing (source, version, asset, etc.)
    ├── lockfile.rs       # plugins.lock read/write
    ├── github.rs         # GitHub API client (release lookup, asset download)
    ├── resolve.rs        # Template resolution, OS/arch detection
    └── verify.rs         # SHA-256 computation and verification
```

### Dependencies

| Crate | Purpose |
|-------|---------|
| `ureq` | HTTP client (lightweight, synchronous, TLS) |
| `serde` + `toml` | Config serialization/deserialization |
| `serde_json` | GitHub API response parsing |
| `sha2` | SHA-256 computation |

### Workspace Changes

```toml
# Cargo.toml (workspace root)
[workspace]
members = [
    "crates/kish-plugin-api",
    "crates/kish-plugin-sdk",
    "crates/kish-plugin-manager",   # New
]
```

`kish-plugin-manager` is an independent binary crate. It does not become a dependency of the kish binary. kish's build size and dependency tree are unaffected.

### Binary Name

The crate produces a binary named `kish-plugin` (configured via `[[bin]]` in its `Cargo.toml`).

## Changes to kish Binary

Changes are minimal:

### `src/plugin/config.rs`

- Add `sha256: Option<String>`, `source: Option<String>`, `version: Option<String>` to `PluginEntry`
- These fields are deserialized from the lock file but not used during plugin loading

### `src/exec/mod.rs`

- Change `plugin_config_path()` to return `plugins.lock` instead of `plugins.toml`

### No Other Changes

`src/plugin/mod.rs` and all other plugin loading/execution code remains unchanged. The lock file's `path`, `enabled`, and `capabilities` fields are the same shape as the existing config format.

## Error Handling

### `kish-plugin sync` Error Cases

| Case | Behavior |
|------|----------|
| Repository not found (404) | Report error, skip plugin |
| Version/release not found | Report error, skip plugin |
| No matching asset for OS/arch | Report error with hint about platform support, skip plugin |
| Network error during download | Report error, delete partial file, skip plugin |
| Local plugin file not found | Report error, skip plugin |
| SHA-256 mismatch on re-download | Report error (possible upstream tampering), skip plugin |
| `plugins.toml` parse error | Fatal error, exit 2 |
| Cannot write lock file | Fatal error, exit 2 |

### Partial Success

- Successful plugins are written to `plugins.lock`
- Failed plugins are reported to stderr with details
- Next `kish-plugin sync` retries failed plugins automatically (they won't be in the lock file)

## Test Strategy

### Unit Tests (`crates/kish-plugin-manager/`)

| Module | Coverage |
|--------|----------|
| `config.rs` | Parse GitHub source, local source, asset template, enabled default, invalid source format error |
| `lockfile.rs` | Read/write round-trip, partial write (success entries only), empty file |
| `resolve.rs` | Template variable expansion (all OS/arch combinations), default template, `{name}` hyphen-to-underscore conversion |
| `verify.rs` | SHA-256 computation, match/mismatch detection, missing file error |
| `github.rs` | API response JSON parsing, asset name matching, tag format fallback (v-prefixed → bare) |

### Integration Tests

- **Full sync flow**: Mock HTTP server (`mockito`) simulating GitHub API responses. Verify download → SHA-256 → lock file generation end-to-end.
- **Partial failure**: 2 plugins, 1 returns 404 → verify only successful plugin appears in lock file.
- **Checksum mismatch**: Existing binary differs from lock file → verify re-download behavior.
- **Local plugin**: `source = "local:..."` with existing/missing file paths.

### kish Binary Tests

- Existing 37 tests in `tests/plugin.rs` continue to pass (lock file loading is compatible with existing `PluginConfig` shape)
- Add test verifying `plugin_config_path()` returns `plugins.lock`

### Not Tested

- Real GitHub API calls (use mocks for CI stability)
- Cross-platform binary compatibility (unit test OS/arch detection logic only)
